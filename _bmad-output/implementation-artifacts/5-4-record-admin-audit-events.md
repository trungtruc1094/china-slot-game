# Story 5.4: Record Admin Audit Events

Status: done

## Story

As an operator,
I want admin actions recorded in an audit trail,
So that operational changes are accountable.

## Acceptance Criteria

- Given an admin changes configuration, limits, manual adjustments, alert acknowledgments, or support-visible records, when the action is completed, then the backend records actor, timestamp, action type, before/after values where applicable, reason, and request ID.
- Audit records cannot be edited through normal admin UI.
- Failed protected actions are logged where security-relevant.
- Tests cover audit creation for representative admin actions.

## Dev Notes

### Unified Audit Event Schema

- Repository contract: append-only `AdminAuditRepository` with `record(input)` and `list()` only; no update/delete API is exposed to routes or domain services.
- Event fields:
  - `id: string` stable audit id generated by the repository.
  - `occurredAt: Date` event timestamp from the shared clock.
  - `actor: string` admin actor or system actor (`alert-service`, `unknown-admin`).
  - `role: "admin" | "operator" | "support" | "viewer" | "system" | "unknown"`.
  - `action: string` dot-delimited action such as `config.activate`, `operator_limits.update`, `alert.firing`, `budget_protection.apply`, `admin.spin_ledger.search`, `admin.balance_transactions.search`, `admin.auth.unauthenticated`, or `admin.auth.forbidden`.
  - `resource: { type: string; id: string }` normalized target, for example `{ type: "config_version", id: draftId }`, `{ type: "operator_limits", id: recordId }`, `{ type: "alert", id: alertId }`, `{ type: "budget_protection_action", id: actionId }`, or `{ type: "admin_search", id: requestId }`.
  - `requestId: string | null` request id when event originated from an API request.
  - `reason: string | null` operator-provided reason where available.
  - `source: "admin-api" | "config" | "operator-limits" | "alerts" | "budget-protection" | "support-search" | "auth"`.
  - `outcome: "succeeded" | "failed"`.
  - `before: Record<string, unknown> | null` relevant prior state or null.
  - `after: Record<string, unknown> | null` relevant resulting state or null.
  - `metadata: Record<string, unknown>` source-specific context with sensitive values minimized.
- Indexes for durable implementation:
  - Primary key on `id`.
  - `(occurred_at desc, id desc)` for default timeline.
  - `(actor, occurred_at desc)` for actor filtering.
  - `(action, occurred_at desc)` for action filtering.
  - `(resource_type, resource_id, occurred_at desc)` for resource filtering.
  - `(request_id)` sparse index for request trace lookups.
  - `(source, occurred_at desc)` for source-specific audits and migrations.
- Migration decision:
  - Existing Epic 3 config activation/rollback emitters are migrated by dual-writing their legacy domain audit records and unified audit events from `InMemoryGameConfigurationRepository`.
  - Existing Epic 4 operator-limit change emitters are migrated by dual-writing legacy limit audit records and unified audit events from `InMemoryOperatorLimitsRepository`.
  - Existing Epic 4 alert fires/resolves/acknowledgments are migrated by writing unified audit events from `InMemoryAlertRepository.appendEvent`.
  - Existing Epic 4 budget-protection apply/revert emitters are migrated by dual-writing legacy budget-protection audit records and unified audit events from `InMemoryBudgetProtectionRepository`.
  - Legacy per-domain audit list endpoints remain for compatibility, but Story 5.5 must query the unified repository only.
- Admin action decision:
  - 5.1 auth failures are recorded as unified `admin.auth.unauthenticated` or `admin.auth.forbidden` events where protected 5.2/5.3 routes reject access.
  - 5.2 spin-ledger searches record `admin.spin_ledger.search` events.
  - 5.3 balance-transaction searches record `admin.balance_transactions.search` events.
- Assumption: this epic continues with in-memory repositories plus production-shaped contracts; durable SQL migration execution is deferred to launch-readiness work.
