## Review Feedback Patterns

- Server-side enforcement was repeatedly required; client-only behavior was not accepted as a launch guardrail.
- Unified audit events remained the canonical evidence trail.
- Public contracts and error cases were documented in story Dev Notes before stories were marked done.
- Non-leaky user-facing errors were treated as launch safety, not UI polish.
- Evidence-based review improved as the epic progressed: code behavior, docs, tests, and sprint/story status all had to agree.
- Final checklist review required concrete file paths, test names, and commit hashes.

## Lessons Learned

1. Launch readiness should be treated as a separate artifact from feature completion. Story status can be done while launch remains blocked pending human review.
2. Compliance-sensitive behavior must be server-enforced before wallet or spin state can mutate.
3. Safe failure is a product feature. A backend outage path that avoids optimistic success protects both trust and auditability.
4. A single correlation ID across request trace, wallet transaction, and audit event creates a practical support trail.
5. Retention policy should be explicit even when destructive jobs remain disabled.
6. CI quality gates should reflect the actual manual gate and pin current coverage baselines rather than make unverified claims.
7. A launch checklist is strongest when it cites concrete evidence, including paths, tests, and commit hashes.
