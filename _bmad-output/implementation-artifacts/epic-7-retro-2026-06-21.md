## Review Feedback Patterns

- Real PostgreSQL behavior repeatedly found issues that in-memory tests would not catch.
- Review findings often targeted transaction ordering, cache timing, idempotency residue, and DB-enforced constraints.
- Production safety required negative tests: missing database URL, unreachable database, schema not ready, missing active config, and no in-memory fallback.
- Non-cash language was checked in every persistence area that touched balances, rewards, provider metadata, or future top-up records.
- Search and support requirements became stronger when data moved from arrays to indexed PostgreSQL queries.
- The final gate favored reconstruction evidence over implementation claims.

## Lessons Learned

1. Durable persistence is a product safety feature, not infrastructure cleanup. It protects balances, operator budgets, retries, support search, and future Tevi integration.
2. Idempotency only becomes trustworthy when it is stored beside the wallet and ledger transaction it protects.
3. Production composition should be explicit and boring. A safe startup failure is better than a quiet in-memory fallback.
4. PostgreSQL integration tests are mandatory for persistence behavior. Unit tests can prove contracts, but not migration readiness, row locking, FK constraints, rollback, or restart recovery.
5. Cache hydration needs to be treated as startup behavior with tests, not an implementation detail left to first request traffic.
6. Future-provider scaffolding must be provider-neutral and non-cash until policy and integration scope are approved.
7. Operational documentation should name gates directly: migration, schema readiness, integration tests, recovery verification, and human acceptance.

## Significant Discoveries

- The original architecture choice of PostgreSQL as authoritative storage was sound, but production composition was the decisive missing piece.
- Recovery verification is most valuable when it crosses repository families. Isolated persistence tests are necessary but not sufficient.
- Durable provider idempotency can be created safely before Tevi work only if the implementation remains provider-neutral and avoids wallet crediting behavior.
- Persistence readiness narrowed the launch blocker list. Durable storage is no longer the broad unknown; production identity, policy acceptance, retention approval, and operational owner decisions are now the primary blockers.
- Fail-safe startup and readiness checks are part of gameplay correctness. A game that starts with the wrong storage backend is not merely misconfigured; it is unsafe for reward-bearing play.
