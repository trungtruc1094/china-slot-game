## Challenges

Charlie (Senior Developer): "The legacy client had several traps: 243-ways behavior was implicit, `Pay` versus `pay` could hide bad ordering, and server examples looked useful but were not canonical."

Dana (QA Engineer): "The most common quality challenge was boundary behavior. Several stories initially passed happy paths but needed stronger tests around invalid shapes, mismatched indexes, malformed paytables, and payout scaling."

Winston (Architect): "The package isolation test was both helpful and brittle. It forced clean boundaries, but source-text scanning means future diagnostics and comments need careful wording."

Amelia (Developer): "The full current-config RTP calculation is expensive enough that routine tests should use compact fixtures for exact math and reserve current-config checks for shape and diagnostics."

## Review Feedback Patterns

- Story 1.1 review tightened package contract, TypeScript non-DOM settings, fixture fidelity, symbol metadata, and isolation checks.
- Story 1.2 review tightened visible-window invariants and configured reel-index handling.
- Story 1.3 review tightened paytable prefix validation, jackpot amount validation, jackpot visible-count reporting, and wild-substitution negatives.
- Story 1.4 review corrected RTP payout aggregation so line-bet multiplication affects report totals and distribution.
- Story 1.5 review strengthened simulation tests to assert exact seeded aggregate output.

Dana (QA Engineer): "The pattern is clear: every review added a durable guardrail. That is a healthy process."

## Lessons Learned

1. Hidden legacy behavior must be converted into explicit package tests before backend work depends on it.
2. Review should focus on edge cases, not just acceptance checklist completion.
3. Math reports and simulations must apply payout policy consistently, especially line-bet multiplication.
4. Compact deterministic fixtures are essential for exact math assertions.
5. Current-client fixtures are best used for fidelity, diagnostics, and shape coverage, not every exhaustive calculation.
6. Package isolation belongs in tests because future API work will be tempted to import the wrong layer.

## Technical Debt and Risks

- No open Epic 1 story debt is recorded as unresolved.
- The game math package does not yet expose a backend spin engine; Epic 2 must wrap the package rather than duplicating it.
- Current active config still has economics and content issues for product tuning: low engagement RTP concern, unreachable `Scroll`, and unused `10` metadata.
- Compliance and reward model decisions remain outside Epic 1 and still block reward-bearing launch.
- `server_examples` remain non-canonical and should not be reused as implementation source.

## Significant Discoveries

- The server example mismatch is not incidental; it is a recurring risk that must be called out in future backend stories.
- Line-bet multiplication is easy to miss when moving from win units to report/simulation totals.
- Exhaustive math can be correct but too heavy for routine current-config tests; compact fixtures are the right testing tool for exact assertions.
- Review findings materially improved correctness and should remain part of the BMAD flow.

## Team Agreements

- Reuse `packages/game-math` for all production spin, RTP, and simulation behavior.
- Prefer deterministic fixtures for math correctness.
- Treat client-side RNG and payout logic as demo-only until backend authority is in place.
- Keep review findings in story records and resolve them before marking done.
- Avoid copying from `server_examples` into production backend logic.
