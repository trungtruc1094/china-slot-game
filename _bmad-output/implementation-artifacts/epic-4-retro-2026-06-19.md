## Completed Scope

- Story 4.1: Added versioned operator limits with explicit unit-bearing limits, validation, admin APIs, audit events, and `operator_limits` migration.
- Story 4.2: Enforced active operator limits during spin validation before reel generation or wallet debit.
- Story 4.3: Added live metrics aggregation from accepted spin ledger rows with UTC window filters, config-version filters, and `operator_metric_buckets` persistence contract.
- Story 4.4: Added alert rules, deterministic alert evaluation, append-only alert history, acknowledgments, alert state in metrics, and alert migration artifacts.
- Story 4.5: Added reversible budget protection actions, feature gating, spin enforcement, audit trail, and budget protection migration artifacts.

## Outcomes

- Operators can configure active limits by scope.
- Spins violating active limits are rejected with stable client-displayable errors.
- Rejected spins do not mutate wallet state, spin ledger state, idempotency acceptance, or reel generation.
- Admin metrics return total wagered, total paid, observed RTP, theoretical RTP, hit rate, player count, active sessions, jackpot liability, remaining budget, and alert state.
- Metrics can be filtered by UTC window and configuration version.
- Alert evaluation is deterministic and idempotent for the same rule/window/config/scope input.
- Alert history is append-only and supports acknowledgment.
- Budget protection actions cover disable paid spins, lower max bet, pause campaign, and require host approval.
- Every budget protection action is reversible through an active/reverted action record.
- Budget protection can be disabled per environment through configuration.
- Final quality gate passed for every story: lint, typecheck, test, and build.

## Review Feedback Patterns

- Public API contracts were required in every story and are now consistently documented.
- Assumptions were made explicit instead of being left in implementation memory.
- Persistence decisions moved from vague intent to concrete schema, retention, and indexing notes.
- Tests repeatedly crossed boundaries: admin writes to spin reads, metrics to ledger rows, alert state to metrics, and protection actions to spin rejection.
- The same launch-readiness gap remains visible: in-memory runtime storage with migration artifacts, not a live database-backed repository layer.

## Lessons Learned

1. Budget control is a chain, not one feature. Limits, enforcement, metrics, alerts, and protection actions have to agree on scope, units, and audit semantics.
2. Future-facing controls are testable. Proving RNG, wallet, and ledger non-mutation is the clearest way to protect the fair-operation guardrail.
3. Metrics need source-of-truth discipline. Accepted spin ledger rows are the right live source; materialized buckets should be replayable from append-only source records.
4. Idempotency is not just for spins. Alert evaluation also needed deterministic keys to avoid duplicate operational noise.
5. Reversibility should be designed into controls from the start. Active/reverted records made budget protection safer and easier to audit.
6. Story files are doing real architectural work. Dev Notes now capture contracts future stories can depend on.

## Technical Debt and Risks

- Runtime storage remains in-memory for operator limits, metrics, alerts, budget protection, configuration, wallet, and ledger.
- Migration artifacts are present but not executed against a live Postgres test harness.
- Admin auth remains header-based scaffolding and must be replaced by Epic 5.
- Audit records are domain-specific and split across repositories; Epic 5 should provide a unified admin audit model/search experience.
- Scope routing defaults to `default`; campaign/config-aware scoping needs hardening.
- Metrics are computed live from in-memory ledger rows; materialized bucket jobs are only specified, not implemented.
- Alert coverage starts with supported metrics from Story 4.3; suspicious activity and backend error-rate alerting remain future observability work.

## Significant Discoveries

- Epic 4 created several domain-specific audit streams. That is useful locally but creates a clear Epic 5 integration requirement.
- Metrics and alerts are only as trustworthy as accepted ledger rows; rejected spins and future-facing protection actions must stay out of accepted outcome metrics.
- Feature flags are not optional for operational controls. Budget protection can now be disabled per environment, and Epic 5 should preserve that operational safety mindset.
- `scopeId = default` is acceptable scaffolding but should not be mistaken for final campaign scoping.
- The existing Epic 5 plan is directionally sound, but Story 5.4 and Story 5.5 should explicitly account for Epic 4's newly added audit sources.

## Readiness Assessment

Testing and Quality: Strong for Epic 4 scope. Story-level tests cover acceptance criteria, and the final quality gates passed with lint, typecheck, test, and build.

Deployment: Not production-ready as a reward-bearing control system until durable repositories, real admin auth, migration execution, and launch retention policies are implemented.

Stakeholder Acceptance: Story acceptance is complete in sprint status. Product acceptance should focus on whether the operator control semantics match host expectations for campaign policy.

Technical Health: Stable for process-local implementation and well documented. The main concern is the number of in-memory repositories modeling production-shaped behavior.

Unresolved Blockers: None for closing Epic 4. Before Epic 5 is complete, the project should unify admin auth and audit search over the operational records introduced here.

## Team Agreements

- Keep all profitability and protection controls explicit, logged, and future-facing.
- Treat in-memory repositories as scaffolding unless a story explicitly promotes them to durable runtime storage.
- Keep API contracts and assumptions in story Dev Notes.
- Require tests that prove rejected or protected paths do not mutate accepted outcomes.
- Reconcile support-facing records against source-of-truth ledger and wallet records.
- Do not start Epic 5 implementation without accounting for the Epic 4 audit sources.
